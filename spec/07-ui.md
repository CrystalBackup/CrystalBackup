# UI — browse & download strategy (R9)

Status: draft v1. Decision record: [adr/0008-ui-strategy.md](adr/0008-ui-strategy.md).
Priority: **lower than everything else** (agreed 2026-07-11, cf.
[00-requirements.md §7](00-requirements.md)); v1 UI lands at milestone M7
([90-roadmap.md](90-roadmap.md)).

## 1. Scope

R9: list the backups of a location, browse a backup's file tree, download individual
files or directories **without retrieving the whole backup**. Two stages:

1. **v1 (M7)** — `crystalctl ui`: a local thick client. A single Go binary (a subcommand of
   the existing `crystalctl` CLI, cf. [06-cli.md](06-cli.md)) that serves an embedded SPA on
   localhost and opens the user's browser — shipped across the **same cross-platform matrix as
   the CLI** (linux/windows/darwin on amd64 + arm64, [06-cli.md §7](06-cli.md)). No server-side
   deployment, no authentication layer, no multi-tenancy problem to solve: the user brings
   their own credentials, exactly like the R8 CLI path.
2. **Long-term (post-v1, separate ADR when the milestone is scheduled)** — a hosted
   multi-tenant backend consumed by a **Rancher extension or Headlamp plugin** (choice
   deferred — §5). Keycloak OIDC; the backend opens repositories server-side, so
   credentials and keys never reach the browser.

Both stages share one Go package (`internal/browse`, §4) — the thick client is the first
consumer, the hosted backend is the second. Everything below is designed so that stage 2
adds an authn/authz layer and a deployment, not a rewrite.

### Non-goals of the v1 UI

- **No restore-into-cluster actions.** Restores are driven by the `Restore` /
  `ClusterRestore` CRs and the CLI; the UI is read-only towards the cluster and the
  repositories. This keeps R23 confirmation flows in one enforcement point (the ValidatingAdmissionPolicy, [adr/0010](adr/0010-admission-vap-first.md)) and
  the UI out of the destructive-action threat model.
- No `BackupSchedule` / location editing, no admin operations, no metrics dashboards
  (Grafana covers that, cf. [05-observability.md](05-observability.md)).

## 2. Why not reuse an existing UI

From the 2026-07-11 survey of existing UIs:

| Candidate | Verdict | Blocking reasons |
|---|---|---|
| Backrest (garethgeorge/backrest) | rejected | **GPL-3.0** (incompatible with the project's permissive-license constraint for an open-source product); **no OIDC** (issue #1123 open, no maintainer commitment); multi-user exists but every user sees every repo — no per-user isolation, unusable multi-tenant. Fine as an internal admin troubleshooting tool, nothing more. |
| Zerobyte (nicotsx/zerobyte) | rejected for v1, **optional POC noted** | **AGPL-3.0** (network copyleft: serving users obliges publishing modifications); v0.x with declared breaking changes. But: organizations + generic OIDC (Keycloak-compatible) + attach existing restic repos + selective restore — the closest existing match. A time-boxed POC before the hosted-backend milestone may re-evaluate it (open question §7); direct in-browser file download is undocumented and must be verified. |
| KopiaUI / `kopia server --ui` (htmlui) | rejected | **Wrong engine**: kopia format, not restic ([adr/0001](adr/0001-repository-engine-restic-format.md)). Additionally its web UI connects with server-control credentials — a global admin view, explicitly not designed for multi-tenant end users. |
| restic-browser (emuell) | out of scope | Desktop Tauri app, single-user, no server component. Admin debugging tool at best. |

Conclusion: nothing ships R9 with per-tenant isolation and Keycloak OIDC on restic
repositories. We build a thin one ourselves, in two stages.

## 3. v1 — `crystalctl ui` thick client (M7)

### 3.1 Process model

- `crystalctl ui` starts an HTTP server bound to `127.0.0.1` on a random free port
  (`--listen 127.0.0.1:PORT` to pin, `--no-open` to skip browser launch), then opens
  `http://127.0.0.1:PORT/?token=<random>` in the default browser.
- The SPA (static assets) is compiled into the binary via `embed.FS` — one artifact, no
  CDN, works air-gapped.
- **Local security**: the server binds loopback only and requires the per-session random
  bearer token minted at startup (passed once in the launch URL, then held by the SPA);
  this blocks other local users and cross-site requests from web pages. Repository
  passwords and S3 credentials stay in the Go process; the browser only ever sees
  listings and file bytes.

### 3.2 Credential and inventory sources

Same inputs as the CLI, two modes usable together — the two views R9 has to serve:

- **Direct repository mode (no Kubernetes — R8-style, the *standalone view*)**:
  `crystalctl` config file / environment provides the repository URL
  (`s3:https://…/<prefix>/<clusterID>`), S3 credentials and the restic password for that
  repo — the **user's own key** for a namespace repository, or the **platform key** for the
  shared cluster DR repository. The UI lists that repository's **restic snapshots**
  directly. This is the standalone path: a user reading their own off-platform backups, or
  an admin reading the shared cluster DR repo when the platform is down.
- **Kubeconfig mode (the *in-cluster view*)**: with a kubeconfig present, the UI presents a
  catalogue built from the **namespaced `Backup` objects** visible to the user's RBAC (plus
  `BackupLocation` / `ClusterBackupLocation` / `BackupRepository` where RBAC allows) —
  schedule names, backup phases, per-volume `snapshotID`s from `Backup.status.volumes`.
  `Backup` objects are namespaced, so RBAC scopes the catalogue to what the user owns
  (native tenant visibility — [02-api.md](02-api.md)). Opening a repository still requires
  the repository password from local config or an interactive prompt: the UI never extracts
  the platform-wrapped cluster DEK, and a user can only read Secrets their RBAC already
  grants (their own namespace's `BackupLocation` password).

### 3.3 Capabilities (v1)

- List configured locations and repositories; show `BackupRepository.status` summary
  (kubeconfig mode, admin RBAC).
- List a repository's snapshots, grouped by `namespace=` / `run=` / `pvc=` tags (snapshot
  tag contract in [02-api.md](02-api.md)).
- Browse a snapshot's file tree with **lazy directory listing** — one request per
  directory opened, never a full-tree fetch.
- Download a single file, **streamed** (no temp file, no size limit beyond the user's
  patience).
- Download a directory as a **zip or tar archive, streamed** as it is produced.
- Show manifest snapshots (`kind=manifests`, path `/manifests/<namespace>`) as browsable
  YAML files — a convenient read-only preview of the resources a `Restore` /
  `ClusterRestore` would apply (the `resources` axis of the
  [restore selection model](02-api.md#restore-selection-model)).

### 3.4 HTTP API (localhost; the future hosted API keeps these shapes)

```
GET /api/v1/repositories
GET /api/v1/repositories/{repo}/snapshots?namespace=&pvc=&run=
GET /api/v1/repositories/{repo}/snapshots/{snapshotID}/ls?path=/var/lib
GET /api/v1/repositories/{repo}/snapshots/{snapshotID}/file?path=/etc/app.conf      (stream)
GET /api/v1/repositories/{repo}/snapshots/{snapshotID}/archive?path=/data&format=tar (stream)
```

Directory listings are immutable once written (a restic snapshot never changes), so
responses to `ls` are served with long-lived cache headers keyed by `snapshotID`.

## 4. `internal/browse` — the shared backend package

One reusable Go package is **the** product of M7, more than the SPA itself. It is the
single implementation of "read a restic-format repo for humans", shared verbatim by the
thick client and the future hosted backend.

```go
// internal/browse
type Browser interface {
    ListSnapshots(ctx context.Context, f SnapshotFilter) ([]Snapshot, error)
    ReadDir(ctx context.Context, snapshotID, path string) ([]DirEntry, error)   // one level, lazy
    OpenFile(ctx context.Context, snapshotID, path string) (io.ReadCloser, error) // streamed
    Archive(ctx context.Context, snapshotID, path string, format ArchiveFormat, w io.Writer) error
}

func Open(ctx context.Context, cfg RepoConfig) (Browser, error) // cfg: repo URL, S3 creds, password, cacheDir
```

### 4.1 v1 implementation: restic CLI subprocess

restic deliberately exposes no Go library (all code under `internal/`; upstream issue
restic#4406 open without commitment), so v1 shells out to the same pinned `restic` binary
already used by the mover image:

- `ListSnapshots` → `restic snapshots --json [--tag …]`
- `ReadDir` → `restic ls --json <snapshotID> <path>` (non-recursive by default — exactly
  the lazy semantic we want; parse the JSON-lines stream)
- `OpenFile` → `restic dump <snapshotID> <path>` — restic streams a single file to
  stdout reading only the needed blobs
- `Archive` → `restic dump --archive tar|zip <snapshotID> <path>` — archive built and
  streamed on the fly, no local materialization

Subprocess hygiene: password via `RESTIC_PASSWORD_FILE` on a private tmpfs path (never
argv), S3 creds via env, `--cache-dir` pinned (§6), context cancellation kills the
process group, stderr captured into structured logs, bounded concurrency (default 2
concurrent restic processes per repository, global cap configurable).

### 4.2 Swappability

The `Browser` interface is the seam. Documented future implementations, no code needed
now:

- **rustic_core** (Rust, Apache-2.0/MIT, same restic format): in-process index, lower
  per-request cost, `webdav` feature as an alternative serving path. Blocked today by its
  own "early development stage, API subject to change" status — revisit alongside the
  ADR 0001 engine revisit data point (M6 bench).
- A long-lived indexer process caching parsed trees across requests, behind the same
  interface, if the hosted backend's load profile demands it.

## 5. Long-term — hosted multi-tenant backend + console plugin (post-v1)

Recorded here as direction; the go/no-go and the frontend choice get their own ADR when
the milestone is scheduled. Design constraints already fixed:

- **Backend**: a Deployment in `crystal-backup-system` wrapping `internal/browse`. Authn =
  Keycloak OIDC (the platform's IdP). The backend resolves the user's namespace set
  **server-side from their cluster RBAC**, not from token claims: for each candidate namespace
  it issues `SubjectAccessReview{verb: create, resource: restores.crystalbackup.io, namespace:
  N}` with the user's identity — **a namespace is visible iff the user may create a `Restore` in
  it** (the same right that lets them actually restore). RBAC is the single source of
  authorization, no bespoke permission model ([03-security-and-tenancy.md](03-security-and-tenancy.md)).
  The catalogue is then the namespaced `Backup` objects in those namespaces.
- **Isolation is structural, mirroring the R2/R14 invariant**: the browser never selects an
  arbitrary repository or namespace. For a **namespace-origin** `Backup` the backend opens
  the user's **own** repository with the user's key; for a **cluster-origin** `Backup` it
  opens the shared DR repo under the **non-forgeable server-side `namespace=<the user's
  namespace>` tag filter** — the same mediation the operator enforces on `Restore`. It
  unwraps the platform DEK (or reads the user key) in-process and holds S3 credentials
  there; **the browser receives listings and bytes, never credentials or keys**.
- **API**: same routes as §3.4 plus the OIDC middleware; per-tenant rate limits and
  archive size caps; access logs and metrics carry the `namespace` label (R19).
- **Frontend**: Rancher extension **or** Headlamp plugin — both remain on the list.
  Rancher is a widely deployed multi-tenant platform console, which weighs in its favor;
  Headlamp (CNCF) hedges against Rancher/SUSE roadmap changes. Both would be thin skins
  over the backend API. Decision deferred.
- Still no restore-into-cluster from the UI in the first hosted iteration; if it ever
  comes, it must go through creating `Restore` CRs with the user's own identity
  (impersonation), never with backend privileges.

## 6. Performance: restic browse latency on S3

Interactive browsing is restic's weak spot: tree blobs live inside pack files, so a cold
`ls` triggers ranged GETs against S3 (the Backrest documentation carries the same
warning — "browsing can be very slow as restic fetches pack files" on remote storage).
Mitigations, in order of impact:

1. **Persistent local metadata cache**: pin `--cache-dir` to a per-repository directory
   that survives across requests and sessions (thick client: under
   `${XDG_CACHE_HOME}/crystalctl/`; hosted backend: a PVC). restic's cache stores index and
   tree packs; after first contact, directory listings are served from disk.
2. **Tree-blob prefetch**: on first snapshot open, warm the cache in the background
   (bounded depth-limited walk of the top directory levels) so that the user's next
   clicks are cache hits.
3. **Immutable-response caching** (§3.4): `ls` results keyed by `snapshotID` are cacheable
   forever at the HTTP layer.
4. Bounded restic-process concurrency to avoid S3 request storms and memory spikes
   (restic loads the repo index per process; the shared cache keeps this cheap after
   warm-up, but large repositories still cost hundreds of MB per process — cap and queue).
