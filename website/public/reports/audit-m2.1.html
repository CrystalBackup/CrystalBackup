M0–M2 Security & Resilience Audit

A read-only sweep of the shipped M0–M2 code — the input to the 0.2.1 hardening patch — across five axes: adequacy to spec, code quality, security (attack / algorithmic / resilience), and multi-tenant isolation.

Scope ~33k LOC Go · 12 CRDs · 16 ADRs Method 5 parallel read-only auditors + direct verification Date 2026-07-19 Outcome 0.2.1
Design sound · one critical implementation defect, all fixed in 0.2.1

The multi-tenant read boundary — the non-forgeable, server-derived namespace= mediation (invariant I1) — and the crypto core (no KEK generation, CSPRNG DEK, age wrap/unwrap, escrow validated-before-trust, secrets never on argv or in logs) were traced end-to-end and found sound. The audit surfaced 1 critical data-integrity defect in the backup fan-out, 3 high-severity correctness/security gaps, and a set of medium/low items — all fixed and re-validated (unit + envtest + lint green; crucible 31/0 on real Ceph).

Deferred (tracked, not in 0.2.1)

Per-repo mover credential scoping (invariant I4 — STS AssumeRole or RGW static per-repo keys, an M6 hardening item); a dedicated tokenless crystal-mover ServiceAccount (movers already run with automountServiceAccountToken: false, so I6 holds — this is defense in depth); an ownerReference/TTL backstop on the maintenance creds Secret; an injectable clock for the orphan reaper. All low-severity and recorded in the CHANGELOG.
