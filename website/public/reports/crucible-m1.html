The crucible runs an independent restic oracle — a throwaway Job running the plain restic CLI against the same S3 repository, wrapped-DEK password and credentials the operator uses. It reads the object store as ground truth, so a controller that both writes and reports the same wrong thing cannot make a check pass. Every assertion below was verified against real Ceph / Longhorn / local-path volumes and Hetzner Object Storage.

Repository & location

3 / 3

A shared cluster repository, created once and safely — the foundation the whole DR plane sits on.

  • A ClusterBackupLocation provisions one initialized shared repository 16.1s

    A location lazily creates and restic inits exactly one repository, reported Initialized.

  • The shared repository is initialized exactly once, even under concurrent reconciles 35.6s

    The per-repo exclusive queue serialises init — no double-init race (K8up #1055) when reconciles overlap.

  • Only one ClusterBackupLocation may be the default 300ms

    The single-default invariant is enforced at admission.

Cascade backup

4 / 4

One schedule fans a cluster-DR run out across six seeded tenants on four storage classes.

  • A ClusterBackupSchedule fans out a Backup into every matched namespace 6m18.3s

    A single schedule stamps one run into all six matched tenant namespaces, linked by label.

  • Every snapshottable PVC lands in the shared repository with the correct restic identityfix #1 20.6s

    Each snapshot carries host + namespace/pvc/kind/run tags and preserves exotic data — xattrs, hardlinks, symlinks (target & type), unicode names.

  • A volume on storage without snapshot support is Skipped, not Failed 100ms

    A local-path PVC (no VolumeSnapshotClass) is neutral in the roll-up — it never degrades the run.

  • A namespace with no PVC completes cleanly 600ms

    A tenant with no volumes reaches a terminal, non-failed phase with zero volume failures.

Reliability & crash recovery

3 / 3

The cascade must converge, leave no litter, and survive both an operator restart and a hard-killed mover.

  • Leak-check — a run leaves zero residual snapshot objects 5m23s

    No VolumeSnapshots, VolumeSnapshotContents, or temporary clone PVCs survive a completed run.

  • The operator killed mid-run converges via Job re-adoption 4m28.4s

    Deleting the operator mid-run re-adopts in-flight mover Jobs (same UID) instead of orphaning or duplicating them.

  • An OOMKilled mover is reported as a failure, not a silent successfix #2 46.1s

    A SIGKILLed mover is surfaced as Failed, and its stale repository lock is force-cleared under the per-repo backup⇄unlock mutex — no leftover lock blocks the next retention forget.

Discovery

3 / 3

Adding a location is enough to make a shared DR repository restorable — with no prior custom resources.

  • Discovery projects a Backup per (namespace, run) into existing namespacesfix #3 2m57s

    Every restore point is projected as a read-only, cluster-origin Backup; a run whose namespace is gone stays reachable only through ClusterRestore.

  • A projected Backup is a view, not the source of truth 13.8s

    Deleting a projected Backup never runs restic forget; discovery re-creates it on its next pass.

  • kubectl get backups lists exactly the restorable set 29.1s

    The projection is a bijection with the repository's run set — no fabricated and no missing entries; forgetting snapshots removes the projection.

Restore

1 / 1

The ultimate contract: the data comes back, byte-for-byte, with nothing but restic.

  • Restores a data snapshot straight from S3 and byte-verifies it against the seed 1m32.2s

    A pure restic CLI — no operator, no CRs consulted — restores a snapshot from S3 and passes sha256sum -c against the seed checksum. Proves the standard restic format & off-cluster recoverability.
