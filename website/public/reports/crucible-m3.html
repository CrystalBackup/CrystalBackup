This is the M3 acceptance gate — the eleven new criteria run under the m3 label against a live cluster. As with every crucible run, it uses an independent restic oracle (a throwaway Job running the plain restic CLI against the same S3 repository, wrapped-DEK password and credentials the operator uses), so a controller that both writes and reports the same wrong thing cannot make a check pass. M3 landed two new capabilities: namespace-manifest backup & restore (the discovery-driven dump, the sanitization engine, and mode-aware apply — Recreate vs SSA-Overwrite) and cluster-scoped DR (cluster-manifest capture with opt-in, selective restore, and the unconditional R23 gate). The 32 skipped lines are the platform / M0 / M1 / M2 checks that belong to other labels and did not run under this filter — a skip is not a failure; those milestones keep their own green reports and non-regression coverage.

M3 · Namespace manifests & DR bootstrap

6 / 6

The first half of M3: capture a namespace’s Kubernetes objects (sanitized) alongside its PVC data, and restore them — reconstituting a namespace, healing config drift under both modes, and honouring the R23 confirmation gate.

  • Bootstraps DR into a created namespace, remaps the storage class, and a workload runs on the restored volume 1m43.4s

    The namespace-DR exit criterion: an absent namespace is reconstituted from the repository alone — manifests and PVC data — into a freshly created target; the source storageClassName is remapped to the target cluster’s class, and a Deployment mounting the restored PVC reaches Running.

  • Captures the namespace manifests: run Completes, child Backup + clusterResourcesCaptured set, kind=manifests snapshot present 53.7s

    The manifests phase dumps the namespace’s discovered resources through the sanitization engine into a kind=manifests restic snapshot; the run reaches Completed and the child Backup’s status records the capture — the oracle confirms the tagged snapshot exists in the repository.

  • Overwrite-restores the drifted ConfigMap via SSA, keeping target-only extras (R23 parks without a confirmation) 15.6s

    Without spec.confirmation the restore parks (R23); once confirmed, Overwrite applies the backed-up manifests with server-side apply — the drifted ConfigMap is healed to the backup, and a key present only in the target but absent from the backup is kept.

  • Recreate-restores the ConfigMap to an EXACT match (target-only extra removed) 15.4s

    Recreate deletes the selected resource and re-creates it from the backup — the ConfigMap matches the snapshot exactly and the target-only key is gone, the manifest analogue of restic restore --delete.

  • Restores the workload manifests: Deployment reaches Ready, nodePort preserved, clusterIP freshly allocated 15.5s

    A restored Deployment rolls out to Ready; the sanitization engine preserves a stable field (the Service nodePort) while dropping a cluster-assigned one (clusterIP), which the apiserver re-allocates — restore-portability without carrying stale cluster state.

  • Rejects a WRONG confirmation at admission (R23 gate) 600ms

    A destructive restore whose spec.confirmation does not equal the target namespace is refused at admission — the R23 gate is unconditional, not a controller-time afterthought.

M3 · Cluster-scoped capture & selective restore

5 / 5

The second half: capture cluster-scoped objects (CRDs, StorageClasses, ClusterRoles…) as their own snapshot, and restore them selectively and opt-in — nothing cluster-wide moves unless explicitly asked, gated by R23 and admin-only RBAC.

  • Captures a kind=cluster-manifests snapshot; clusterResourcesCaptured is set 53.6s

    A cluster-scoped backup dumps the allow-listed cluster resources into a distinct kind=cluster-manifests restic snapshot and stamps clusterResourcesCaptured on status — the oracle confirms the tagged snapshot in the repository.

  • Selectively restores ONLY the included CRD + StorageClass, leaving the un-included ClusterRole absent 20.7s

    A cluster-scoped restore with an explicit selection applies exactly the included CRD and StorageClass; a ClusterRole present in the snapshot but outside the selection is never created — selection is a strict allow-list, not a best-effort filter.

  • Restores NOTHING cluster-scoped when clusterResources is omitted (opt-in default) 10.4s

    Omitting the cluster-resource selection restores nothing cluster-wide — the blast radius of cluster-scoped objects is opt-in by construction, so a namespace restore can never silently mutate shared cluster state.

  • Denies a destructive cluster-scoped restore with a WRONG confirmation (R23 gate) 0s

    The R23 confirmation gate applies to the cluster-scoped path too — a destructive ClusterRestore with a mismatched confirmation is refused, closing the gate on the highest-blast-radius operation in the system.

  • Denies a cluster-scoped restore to a NON-admin identity (RBAC) 700ms

    Cluster-scoped restore is admin-only: a non-admin identity attempting a ClusterRestore is denied by RBAC — the tenancy boundary holds even for a caller who authors a valid spec.

What real RKE2 / Canal surfaced

Three findings from live infrastructure that kind couldn’t — harness & ops, not product defects

M3’s functional code passed envtest and the make e2e kind suite (25/25) before it ever reached the cluster. What only a real RKE2 / Canal / Ceph cluster surfaced was operational: the interaction of M3’s new default-deny NetworkPolicies with a CNI that enforces egress after Service DNAT, and shared-repository hermeticity under a re-used cluster. None were correctness defects in the product — each is a harness or deployment-configuration adjustment, captured here so an RKE2 operator hits none of them.

ops · CNI

Canal evaluates egress post-DNAT

The manifest-mover’s NetworkPolicy must reach the API server. On RKE2, the kubernetes Service (10.43.0.1:443) DNATs to the masters on :6443, and Canal evaluates the egress rule after DNAT — so the allow must name the post-DNAT port. Deploy with networkPolicy.apiServerPort=6443 on RKE2-class clusters.

test harness

Default-deny caught the restic oracle

M3’s default-deny (correctly) blocks all egress by default. The crucible’s independent restic oracle is a Job outside the mover’s path, so it too was denied its S3 egress. Fixed in the harness by tagging the oracle Jobs with the mover-egress label the policy allows.

test harness

Shared-repo hermeticity

On a restic repository re-used across runs, discovery projects the first snapshot for a given backup name and short-circuits duplicates — so a re-used backup name silently skipped a manifest capture. Fixed by deriving per-run-unique backup names in the suite.
